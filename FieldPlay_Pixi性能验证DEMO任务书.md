# FieldPlay · PixiJS 性能验证 DEMO 任务书

> 版本 V0.1 · 2026-06-22 · 面向 CODE（AI）执行
> 目标：用 **PixiJS（WebGL）** 1:1 复刻现有 DOM/CSS DEMO 的**画面与动效**，在**同一部手机**上与 DOM 版对比帧率，验证 WebGL 路线能否消除掉帧。

---

## 〇、目标与范围（先把"为什么做"说清楚）

**唯一变量 = 渲染方式（DOM/CSS → PixiJS/WebGL）。** 其它（美术、布局、动效观感、舞台尺寸、运动节奏）全部对齐现有 DEMO，否则对比不公平。

**做（与画面/性能相关的全做）：**
- 天气×时段背景交叉淡入、12 地块、作物按阶段生长、机器人沿路移动 + 车灯光池、昼夜 tint、浇水/施肥粒子、杂草/极端天气状态叠加。
- 内置 FPS/内存表、效果开关面板、压力模式（用于测量与定位开销）。

**先不做（这是渲染性能验证，不是完整游戏）：**
- 完整业务 UI（商店/订单/社交/账号）、后端、真实经济与 AI 逻辑。
- 以上一律用**假数据驱动画面**即可——目的是让画面和 DEMO 一样"繁忙"，不是跑真实业务。

**成功标准：** 在目标手机上，**同等画面**下 Pixi 版压力档帧率明显高于 DOM DEMO，且无内存崩溃、首屏可接受。

---

## 一、技术栈与工程

- **PixiJS v8 + TypeScript + Vite**，纯前端单页，浏览器运行（手机可直接访问 URL）。
- **复用仓库 `assets/` 同一套美术**（保证画面可比）：36 背景 `bg/bg_<set>_<node>.jpg`、作物 `<plant|crop>_<crop>_s1..5.png`、状态 `state/*`、杂草 `wd/weed_<node>.jpg`、`soil.png`。
- **固定 1280×720 设计舞台** + `fitBoard` 等比缩放（`scale = min(W/1280, H/720)`）；移动端竖屏自动 `rotate(90°)`（与原型一致）。
- 右上角常驻 **FPS / 内存表**（stats.js 或基于 Pixi Ticker 自实现）。

---

## 二、场景搭建（对齐原型 z 序）

分层（低→高）：

1. **背景层**：天气×时段全屏图，交叉淡入。
2. **地块层**：12 个等距地块（用仓库 `BASE_CORNERS`，% of 1672×941 定位）+ 每株作物精灵（按 stage 换贴图）。
3. **状态/杂草叠加层**：`state/*`、`wd/*`。
4. **机器人 + 车灯光池层**。
5. **昼夜 tint 层**。
6. **粒子层**（浇水/施肥）。
7. **最简 HUD**（仅放：模式切换、天气切换、昼夜滑块、效果开关、压力模式按钮——够测试即可）。
8. **FPS/内存表**。

资源管线：作物/状态小图打包成**图集（spritesheet）**以合批；背景**按需加载/解码**，只热"当前 + 过渡"2–4 张（严禁一次性解码 36 张）；背景**按设备分辨率选尺寸**。

---

## 三、必须复刻的"重效果"（DOM 掉帧的根源，正是要在 WebGL 里重做并对比）

| 效果 | 原型（DOM，贵） | Pixi（WebGL，做法） |
|---|---|---|
| **车灯光池** | `mix-blend-mode: color-dodge` + 径向渐变 + `mask`，跟机器人**每帧移动**（整屏重合成） | Sprite/Graphics + 混合模式 `add`/`screen` 近似辉光（可选自定义 filter 精确还原 color-dodge）+ 径向渐变纹理 + `mask`，跟随机器人 `tween` |
| **背景天气×时段交叉淡入** | 多层全屏图 `opacity` 过渡 | 两张全屏 Sprite 做 alpha 交叉淡入；只解码当前+下一张 |
| **昼夜 tint + 夜间车灯** | 半透明叠加层 + filter | 整屏 tint Sprite 或 `ColorMatrixFilter`，由 `tod`(0–1) 驱动 |
| **浇水/施肥粒子** | DOM 节点动画（重排重绘） | `ParticleContainer` / 精灵对象池 |
| **杂草/极端天气叠加** | `state` png 叠加 | 同图集 Sprite 叠加 |
| **作物生长** | 换 `img src` | 换贴图/换帧 |

> 关键认知：这些效果在 DOM 里逼浏览器做**昂贵的合成/重排/重绘**，在 WebGL 里是 GPU 的**家常便饭**。本 DEMO 就是要把同样的效果用 WebGL 实现，让差距可量化。

---

## 四、动态与交互（假数据驱动，不接真实逻辑）

- **tick 循环**驱动画面"动起来"：作物生长（stage 0→4 换贴图）、天气切换（触发背景交叉淡入）、昼夜（`tod` 驱动 tint）、机器人沿路点移动（光池跟随）。
- **交互（够测试即可）**：点地块 = 浇水（触发粒子）；切换天气；拖昼夜滑块；开/关托管（机器人开始/停止巡田）。
- 数值用假数据/随机，**不需要真实经济与 AI**——目的只是让画面和 DEMO 一样繁忙。

---

## 五、性能纪律（内建，必须做）

- 图集合批、`ParticleContainer`、对象池、背景按需解码、按分辨率选背景尺寸、避免每帧 `new` 纹理/对象。
- **效果开关面板**：可单独开关「光池 / 背景淡入 / 粒子 / 昼夜 tint」，用来定位各效果开销。
- **压力模式**：一键拉满（大量作物同时生长 + 连续天气切换 + 机器人移动 + 粒子齐发），模拟 DEMO 最忙状态。

---

## 六、测量与对比协议

在**同一部（建议 2–3 部覆盖中端安卓 + iPhone）真机**上，打开 Pixi 版与 DOM DEMO，各跑三档并记录：

| 端机 | 场景档 | DOM DEMO FPS | Pixi FPS | 内存 | 首屏 | 备注 |
|---|---|---|---|---|---|---|
| 例：中端安卓 | 静置 | | | | | |
| | 常规（生长+天气+机器人） | | | | | |
| | 压力模式 | | | | | |
| iPhone | …三档… | | | | | |

三档定义：**静置**（仅背景+静态地块）／**常规**（tick 全开、机器人巡田、偶发天气）／**压力**（一键拉满）。

**通过标准：** Pixi 版在**压力档**帧率明显更高、无卡死、无内存崩溃。

## 八、给 CODE 的执行步骤（有序）

1. 起 **Vite + TS + Pixi v8** 工程，装 stats.js。
2. 接入仓库 `assets/`，把作物/状态小图打成图集；实现 `fitBoard`（1280×720 舞台缩放）+ 移动端竖屏旋转。
3. **背景层**：天气×时段两图交叉淡入 + 按需解码（只热当前+下一张）。
4. **地块层**：按 `BASE_CORNERS` 摆 12 地块 + 作物精灵（按 stage 换贴图）。
5. **机器人 + 车灯光池**：`add`/`screen` 混合 + 径向渐变纹理 + `mask` + 跟随机器人 `tween`。
6. **昼夜 tint**（`ColorMatrixFilter`/整屏 tint，由 `tod` 驱动）+ **粒子**（浇水/施肥，`ParticleContainer`）。
7. **tick 驱动**（生长/天气/昼夜/机器人，假数据）。
8. **FPS/内存表** + **效果开关面板** + **压力模式**。
9. 自测三档帧率，填好第六章对比表。

## 九、参数默认值（可改）

- **目标机型**：默认中端安卓 + iPhone 各一（用于定背景分辨率档）。
- **光池还原方式**：默认先用 `add`/`screen` 近似辉光（更快）；若你要严格还原 `color-dodge`，再上自定义 filter。
- **作物压力数量**：默认压力模式 ≥ 现 DEMO 同屏株数（确保不低于 DEMO 负载）。

> 一句话给 CODE：**画面、动效、节奏照现有 DEMO 还原，唯独把渲染从 DOM/CSS 换成 Pixi/WebGL，并内建 FPS 表与压力模式，让手机上的差距可量化。**
